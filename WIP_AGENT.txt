# ES|QL - Remove LIMIT limitation for full text functions
# GitHub Issue: https://github.com/elastic/elasticsearch/issues/127415

================================================================================
                              ISSUE SUMMARY
================================================================================

## The Problem
Currently in ES|QL, full text functions (like MATCH, QSTR, etc.) cannot be 
used after a LIMIT clause.

Example of what's blocked:
```esql
FROM index
| LIMIT 100
| WHERE MATCH(field, "search term")   <-- ERROR
```

## Why This Limitation Existed
Full text functions previously could only be executed by Lucene (pushed down 
to the storage/search layer). They couldn't run in the ES|QL compute engine 
(the layer that processes data after it's retrieved).

Since LIMIT is applied during compute, and full text functions had to run in 
Lucene *before* compute, you couldn't use them after LIMIT.

## What Changed
Full text functions can now be executed at the compute engine level — meaning 
they no longer *require* being pushed down to Lucene.

## The Enhancement
Remove the artificial restriction that prevents full text functions from being 
used after LIMIT, since the technical limitation no longer applies.

                  THE FIX (What Issue #127415 Proposes)
================================================================================

┌─────────────────────────────────────────────────────────────────────────────┐
│                      EXECUTION MODEL (After Change)                         │
└─────────────────────────────────────────────────────────────────────────────┘

   Full Text Functions can NOW run in COMPUTE ENGINE (not just Lucene)

                    ┌─────────────────────┐
                    │    COORDINATOR      │
                    │                     │
                    │  1. LIMIT (100)     │  ◄── Data reduced to 100 rows
                    │                     │
                    │  2. MATCH can run   │  ◄── NEW! Runs in compute engine
                    │     HERE now!       │      (doesn't need Lucene anymore)
                    │                     │
                    └─────────────────────┘

   Since MATCH/QSTR/etc. can now execute at compute level,
   the restriction in FullTextFunction.java:278 can be REMOVED:

   BEFORE:  lp -> (lp instanceof Limit == false)   // blocks Limit
   AFTER:   lp -> true                             // allow Limit (remove check)

## checkCommandsBeforeExpression method (lines 323-345)

```java
private static <E extends Expression> void checkCommandsBeforeExpression(
    LogicalPlan plan,
    Expression condition,
    Class<E> typeToken,
    Predicate<LogicalPlan> commandCheck,
    java.util.function.Function<E, String> typeErrorMsgProvider,
    Failures failures
) {
    condition.forEachDown(typeToken, exp -> {
        plan.forEachDown(LogicalPlan.class, lp -> {
            if (commandCheck.test(lp) == false) {
                String sourceText = lp.sourceText();
                String errorMessage = sourceText.split(" ")[0].toUpperCase(Locale.ROOT);
                // ...
                failures.add(fail(plan, "{} cannot be used after {}", typeErrorMsgProvider.apply(exp), errorMessage));
            }
        });
    });
}
```

================================================================================
                           POTENTIAL FIX APPROACH
================================================================================

To remove the LIMIT restriction:

1. Modify FullTextFunction.java:274-284
   - Remove `(lp instanceof Limit == false)` from the predicate

2. Similarly for Score function check at lines 290-297
   - Remove `(lp instanceof Limit == false)` from the predicate

3. Update/remove related tests in VerifierTests.java that expect the error

4. Add new tests to verify full text functions work correctly after LIMIT

5. Consider edge cases:
   - Performance implications
   - Scoring behavior after LIMIT
   - Interaction with other pipeline breakers
